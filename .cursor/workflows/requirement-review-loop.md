# 要件定義 → ディベートループ

AIアシスタント（Cursor等）が「設計者」と「QAレビュアー」の2役に分かれ、要件定義書をもとに基本設計書の品質を自律的に高めるワークフローです。
議論プロセスはエディタ上で可視化され、最終的に合意された設計書が出力されます。

## 前提条件

- [設計者用] `docs/templates/requirements-template.md` または `docs/api/spec-api.md`, `docs/design/spec-db.md` のフォーマットを把握済み
- [レビュアー用] セキュリティ、エッジケース、矛盾点の3点を重点的にチェックする

## ワークフロー

### Step 1: 要件定義の提示（ユーザー）

ユーザーがチャット等で「以下の要件定義で基本設計を行ってください: ...」と指示を出します。

### Step 2: Multi-Agent Debate プロセスの開始（Cursor）

Cursorエディタ内で、`.cursor/rules/debate-design.mdc` に定義されたルールに従って以下のプロセスを実行します。

1. **要件解釈**: まず、与えられた要件を確認。
2. **Draft作成 (Agent A)**: Agent Aが初期の設計書ドラフトを提示。
3. **批判的レビュー (Agent B)**: Agent Bがドラフトに対して指摘・懸念（セキュリティ、エッジケース、矛盾）を提示。
4. **自己修正 (Agent A)**: Agent Aが指摘を受け入れ、修正案を提示。
5. **合意形成**: 指摘事項がなくなるか、数ターン（最大3回程度）の議論を経て合意に至るまで自己修正を繰り返す。

### Step 3: 設計書の出力（Cursor）

1. 議論の結果合意された、完全な設計書（Markdown形式）を出力（または対象ファイルに追記・修正）します。
2. その際、`TODO`, `TBD`, `FIXME`, `未定` などの禁止ワード（プレースホルダー）を残さないよう自己検証します。

### Step 3.5: ADR生成（Cursor）

設計書の確定に伴い、以下の重要判断についてADRを生成します：

1. 採用したライブラリ・フレームワークの選定理由
2. データベース設計で選んだアプローチの根拠
3. セキュリティ実装方針など
4. `docs/architecture/decisions/ADR-{次番号}-{slug}.md` としてフォーマットに従い出力
5. `docs/architecture/decisions/README.md` のインデックスを更新

### Step 4: 人間のレビューと完了処理

出力を確認し、問題なければコミット・Pushします。
Git Push 時に、GitHub Actions の `check-ai-laziness` ジョブがプレースホルダーの混入を防ぐ品質ゲートとして機能します。
