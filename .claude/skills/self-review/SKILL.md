# self-review スキル

一発目の出力は下書きとして扱い、提出前に必ず一度「批評者の目」で見直す。
code-reviewer SubAgent（独立コンテキストでの厳格レビュー）とは別物であり、その**前**に実行するメインスレッドの軽量パス。

## Step 1: 要求の再読

元の指示・仕様書を**もう一度読み直し**、要求事項を列挙して成果物と突き合わせる:

- [ ] すべての要求に対応したか（対応できなかった項目は明示的に報告リストへ）
- [ ] 指示にない余計なことをしていないか（スコープクリープ・無断のリファクタ）

## Step 2: 差分の全量確認（コードの場合）

```bash
git diff
git status --short
```

- [ ] 意図しない変更・デバッグ残骸（console.log・コメントアウト・一時ファイル）が混入していないか
- [ ] 変更が計画（deep-plan の計画書）と一致しているか。乖離があれば計画書を更新したか

## Step 3: 品質チェックリスト

**コード:**

- [ ] エッジケース（空・null・上限・権限なし・同時実行）を扱ったか（`core-backend.md`）
- [ ] エラーハンドリングは規約通りか（カスタムエラー・握りつぶしなし）
- [ ] セキュリティ観点（秘密情報・インジェクション・認可漏れ）に問題ないか（`core-security.md`）
- [ ] 命名・パターンがプロジェクト規約（`project-stack.md`）に従っているか
- [ ] テストを同一変更セットで書いたか。テストは境界値・異常系を含むか

**ドキュメント・設計書:**

- [ ] テンプレート・フロントマター・必須セクションに準拠しているか（`core-docs.md`）
- [ ] 曖昧な表現が残っていないか（数値・条件・振る舞いが具体化されているか）
- [ ] 関連文書（要件 ↔ API ↔ DB）との整合性は保たれているか

**共通:**

- [ ] 実行して確認していないのに「動く」と書いていないか（未検証事項は明示）
- [ ] ドキュメント同期（spec-api / spec-db）を忘れていないか

## Step 4: 修正と報告

1. 指摘が見つかったら**報告前に修正**する（修正後、影響があれば再検証する）
2. 完了報告に「セルフレビューで発見・修正した事項」を1〜3行で含める（なければ「指摘なし」と記載）

## 位置づけ

```text
実装 → build-verifier（機械検証）→ self-review（このスキル）→ code-reviewer（独立レビュー）→ 完了報告
```
