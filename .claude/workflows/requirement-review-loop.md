# 要件定義 → ディベートループ

Claude Code が「設計者」と「QAレビュアー」の2役に分かれ、要件定義書をもとに基本設計書の品質を自律的に高めるワークフロー。
議論プロセスは会話上で可視化し、最終的に合意された設計書を出力する。

## ペルソナ定義

### Agent A（設計者）

- **役割**: 大規模 Web アプリケーションのシニア・システムアーキテクト。要件を正確に読み取り、スケーラビリティと保守性を重視した設計を行う
- **ミッション**: 要件定義をもとに実装に必要な設計書を抜け漏れなく作成する。Agent B の指摘は素直に受け入れ、論理的に修正案を提示する

### Agent B（レビュアー）

- **役割**: 厳格な QA リード兼セキュリティ・エンジニア。設計の「穴」を見つけるプロフェッショナル
- **ミッション**: 肯定的なフィードバックは最小限にし、以下の3点に絞って徹底的に指摘する
  1. 要件定義と設計の間の矛盾点
  2. エラー処理・例外状態（エッジケース）の考慮漏れ
  3. セキュリティ上の懸念点

**指摘の重大度基準:**

- `Critical` — セキュリティ脆弱性・データ破壊リスク。次工程進行不可
- `Major` — 要件矛盾・エッジケース漏れ。修正後に再レビュー必須
- `Minor` — 表現の曖昧さ・補足推奨。修正任意

## ワークフロー

### Step 1: 要件定義の提示（ユーザー）

ユーザーが「以下の要件定義で基本設計を行ってください: ...」と指示を出す。

### Step 2: Multi-Agent Debate（Claude Code）

1. **要件解釈**: 与えられた要件を確認する
2. **Draft 作成（Agent A）**: 初期の設計書ドラフトを提示する
3. **批判的レビュー（Agent B）**: ドラフトに対して指摘・懸念（矛盾・エッジケース・セキュリティ）を重大度付きで提示する
4. **自己修正（Agent A）**: 指摘を受け入れ、修正案を提示する
5. **合意形成**: Critical / Major 指摘がなくなるまで 3〜4 を繰り返す（最大3ラウンド程度で打ち切り可）

議論の各ラウンドは会話上で可視化すること。

### Step 3: 設計書の出力（Claude Code）

1. 合意された完全な設計書（Markdown）を出力する（または対象ファイルに反映する）
2. `.claude/rules/core-docs.md` に従い、テンプレート準拠・フロントマター・禁止ワード不使用を自己検証する
3. `npx markdownlint-cli "docs/**/*.md" --fix` を実行する

### Step 3.5: ADR 生成（Claude Code）

設計確定に伴う重要判断（ライブラリ選定・DB 設計アプローチ・セキュリティ実装方針等）があれば:

1. `docs/architecture/decisions/ADR-{次番号}-{slug}.md` をテンプレートに従って作成する
2. `docs/architecture/decisions/README.md` のインデックスを更新する

### Step 4: 人間レビュー（品質ゲート①）

ユーザーが設計書を確認する。

- **承認** → 実装フェーズ（`implement-and-verify.md`）へ
- **差し戻し** → 指摘内容を反映して Step 2 のディベートループから再実行する。設計判断に関わる大きな差し戻しの場合は `rework-report` スキルでエスカレーション記録を保存する
