# アーキテクチャ・設計原則

> [INFO] 汎用コアルール。言語・フレームワーク非依存の設計原則。

## 基本原則

- **関心の分離**: プレゼンテーション・ビジネスロジック・データアクセスを混在させない。依存は常に外側（UI）から内側（ドメイン）へ向ける
- **SOLID**: 特に単一責任（1モジュール1責務）と依存性逆転（具象ではなく抽象に依存）を重視する
- **KISS / YAGNI**: 現在の要件に必要な最小の設計を選ぶ。将来の拡張性のためだけの抽象化を先回りして導入しない
- **DRY は慎重に**: 偶然似ているだけのコードを共通化しない。「同じ理由で変更されるもの」だけを共通化する

## デザインパターンの適用指針

パターンは「問題が現れたときに適用する」もの。パターンを使うこと自体を目的にしない。
適用時は、なぜそのパターンを選んだかをコメントまたは ADR に残すこと。

| パターン | 使いどころ |
| --- | --- |
| Factory / Builder | 生成ロジックが分岐・複雑化したとき |
| Strategy | 同一操作の複数アルゴリズムを切り替えるとき（if/else の増殖の置き換え） |
| Adapter | 外部 API・ライブラリの型を内部の型に閉じ込めるとき |
| Repository | データアクセスをドメインロジックから隔離するとき |
| Observer / Pub-Sub | 疎結合なイベント通知が必要なとき |
| Decorator | 既存処理に横断的関心事（ログ・キャッシュ等）を重ねるとき |

- [NG] Singleton の乱用（グローバル可変状態の温床）。DI やモジュールスコープで代替する
- [NG] 継承の深い階層。合成（composition）を優先する

## モジュール構成

- Feature-based 構成を基本とする: 機能単位でディレクトリを切り、UI とロジックを近くに置く
- 共有部品は共通ディレクトリへ昇格させる（3箇所で使われたら昇格を検討）
- 循環依存を作らない。作りそうになったら層の設計を見直す

## コメント規約（リーダブルコード準拠）

- **Why を書く**: コードから自明な What はコメントしない。背景・意図・前提条件を書く
- 複雑なロジック・非自明な処理・副作用には必ずコメントを付ける
- エクスポートされる関数・公開 API にはドキュメンテーションコメントを付ける
- コード変更時はコメントも同時に更新する（腐ったコメント禁止）
