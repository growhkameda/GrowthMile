# ドキュメント規約

> [INFO] 汎用コアルール。要件定義・仕様書は「AI が再読込して自律的にコードを書けるプロンプト」として機能することを前提に書く。

## 品質基準

- **絵文字禁止**: MD / MDC ファイルでは絵文字を使わず、`[OK]` `[NG]` `[WARN]` `[INFO]` `[POINT]` のプレーンテキストタグを使用する
- 仕様は具体化する: 曖昧な表現を避け、数値・条件・振る舞いを明記する
- 日本語で記述する。技術用語・固有名詞は英語可
- 未解決プレースホルダー・タスクリスト系の禁止ワードを最終出力に残さない（CI が自動検知する）

## 構造

- 新規作成時は `docs/templates/` の該当テンプレートのフォーマット・セクション構造に従う
- 各設計書の冒頭に YAML フロントマター（title, project, version, date, author）を付ける
- エンドポイント一覧・テーブル一覧・パラメータ定義（型・必須/任意・説明）は Markdown 表で書く
- JSON 例は `json`、図は `mermaid` の言語指定付きコードブロックで書く
- Mermaid 図は GitHub / VSCode 上で直接描画されるため、画像（PNG 等）への変換は不要

## 必須セクション

| ドキュメント | 必須セクション |
| --- | --- |
| 要件定義書 | 概要 / スコープ / 機能要件 / 非機能要件 / 更新履歴 |
| API 仕様書 | エンドポイント一覧 / リクエスト・レスポンス例（JSON） / エラーコード一覧 / 認証方式 |
| DB 設計書 | テーブル一覧 / カラム定義（型・制約・説明） / リレーション / インデックス設計 |

## 整合性

- 設計書間（要件 ↔ API ↔ DB ↔ UI）で矛盾を作らない。1つを変更したら関連文書もあわせて更新する
- 実装で仕様が変わった・詳細化された場合は、設計書を同一の変更セットで更新する

## Markdown Lint（CI 対策・必須）

- ドキュメント作成・更新後は必ず `npx markdownlint-cli "docs/**/*.md" --fix` を実行する
- **MD040**: コードブロックには必ず言語名（`text`, `json`, `bash` 等）を指定する
- **MD036**: 見出しの代わりに太字だけの行を作らない
- リスト・見出しの前後には空行を入れる
