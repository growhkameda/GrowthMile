# テスト戦略

> [INFO] 汎用コアルール。テストランナー・コマンドはプロジェクト側ルールを参照。

## テストピラミッド

- ユニットテスト（多）→ 統合テスト（中）→ E2E（少）の比率を保つ
- ビジネスロジックはユニットテストで網羅し、E2E は主要ユーザーフローの確認に絞る

## テストの書き方

- AAA（Arrange-Act-Assert）構造で書く
- 1テスト1検証観点。テスト名は「何をしたら何になる」が読めるように付ける
- 境界値・異常系（空・null・上限超過・権限なし）を必ず含める
- テストデータに実在の個人情報を使わない

## テストの独立性

- テスト間で状態を共有しない（実行順序に依存させない）
- 外部 API はモックする。DB を使う統合テストは各テストでデータを初期化する

## E2E の安定化

- [NG] `waitForLoadState('networkidle')` 等のネットワーク静穏待ちは使わない（CI で長時間ブロックする）
- [OK] URL・要素の表示をポーリングする明示的なアサーションを使う（例: `expect(page).toHaveURL(/\/login/, { timeout: 10000 })`）
- フレーキーなテストは「無効化」ではなく原因を修正する

## 実装とテストの同時性

- 機能実装とテストは同一の変更セットで書く（後回しにしない）
- バグ修正時は、まず再現するテストを書いてから修正する
