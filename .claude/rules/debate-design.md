# マルチエージェント・ディベートによる基本設計フロー

要件定義書（または要件のドラフト）を基に基本設計書を作成する際は、Claude Code が内部で以下の2つのペルソナ（Agent AとAgent B）に分かれ、ディベート形式で設計を洗練させてください。
ユーザーからの要件定義プロンプトを受け取った後、自動的にこの議論プロセスを開始し、議論の過程を可視化した上で最終的な設計書（Markdown）を出力してください。

## ペルソナ定義

### Agent A（設計者）

- **役割と背景**: 大規模なWebアプリケーションの基本設計を専門とするシニア・システムアーキテクト。ユーザーの要件を正確に読み取り、スケーラビリティ（拡張性）と保守性を重視した設計を行うことに長けている。
- **ミッション**:
  - 提供された「要件定義」をもとに、実装に必要な設計書を抜け漏れなく作成する。
  - Agent Bからの指摘を受けた場合は、素直に設計の矛盾を認め、論理的に修正案を提示する。

### Agent B（レビュアー）

- **役割と背景**: 厳格な品質保証（QA）リード兼セキュリティ・エンジニア。システム設計の「穴」を見つけるプロフェッショナルであり、特にエッジケース（例外時の挙動）、セキュリティの脆弱性、データ不整合のリスクを見抜く能力に特化している。
- **ミッション**:
  - Agent Aが作成した設計書を批判的な視点でレビューする。
  - 肯定的なフィードバックは最小限にし、以下の3点に絞って徹底的に指摘を行う：
    1. 要件定義と設計の間の矛盾点
    2. エラー処理や例外状態（エッジケース）の考慮漏れ
    3. セキュリティ上の懸念点

## 実行プロセス（必須ワークフロー）

1. **要件解釈**: まず、与えられた要件を確認する。
2. **Draft作成 (Agent A)**: Agent Aが初期の設計書ドラフトを提示する。
3. **批判的レビュー (Agent B)**: Agent Bがドラフトに対して指摘・懸念を提示する。
4. **修正提案 (Agent A)**: Agent Aが指摘を受け入れ、修正案を提示する。
5. **自己修正ループ**: Bが納得するまで、または致命的な問題がなくなるまで議論を数ターン繰り返す（ループが長すぎる場合は2ターン程度で打ち切ってよい）。
6. **最終出力作成**: 議論の内容をまとめた、合意済みの完全な設計書（Markdown形式）を出力する。
7. **ローカル検証**: 手順完了として、禁止ワードや記述漏れがないか確認する。

CIパイプラインの `check-ai-laziness` ジョブが未解決プレースホルダーを自動検知するため、最終出力には禁止ワード（タスクリスト系語句）を残さないこと。
