# ドキュメント生成の厳格化

要件定義や仕様書（Markdown）の生成・編集時は、Claude Code が再読込して自律的にコードを書ける「プロンプト」として機能することを前提とする。

## 品質基準

- **絵文字の禁止**: MD/MDCファイルにおいて、チェックマークや警告などの絵文字（Emojis）は使用**不可**。AIのトークン効率悪化と意味的ノイズを防ぐため、常に `[OK]`, `[NG]`, `[WARN]`, `[INFO]`, `[POINT]` などのプレーンテキストのタグを使用すること。
- 禁止ワード（タスクリスト系語句）を使用しないこと — CI（`check_ai_laziness` ジョブ）がドキュメント全体をgrepで検査する
- 仕様は必ず具体化すること（曖昧な表現を避ける）
- マークダウンのフォーマットを厳守すること
- 日本語で記述。技術用語・固有名詞は英語可

## 必須セクション

設計書の種類ごとに以下のセクションを必ず含めること：

### 要件定義書（requirements.md）

- `## 概要` — プロジェクトの目的と背景
- `## スコープ` — 対象範囲と除外範囲
- `## 機能要件` — 具体的な機能一覧
- `## 非機能要件` — パフォーマンス、セキュリティ、可用性
- `## 更新履歴` — 変更日、変更者、変更内容

### API仕様書（spec-api.md）

- エンドポイント一覧（メソッド、パス、説明）
- リクエスト/レスポンスの具体例（JSONサンプル付き）
- エラーコード一覧と説明
- 認証方式の説明

### DB設計書（spec-db.md）

- テーブル一覧と各テーブルの目的
- カラム定義（名前、型、制約、説明）
- リレーション（外部キー関係）
- インデックス設計

## 設計の整合性

- 設計書を作成・更新する際は `docs/agentic/mvp_requirements.md`（要件定義）、`docs/rules.md`（技術ルール）の内容を完全に満たすこと
- データベース設計、API仕様、UI/UX、ユーザー体験などの各種要件も、設計書間で相互に矛盾しないよう整合性を保つこと
- 他の設計書を変更したことで整合性が崩れる場合、関連する設計書もあわせて更新すること

## Mermaid図の更新ルール

- 設計書内のMermaid図（ER図、画面遷移図、システム構成図、ワークフロー図等）を変更した場合、対応するPNG画像の再生成が必要
- PNG生成はCIの `generate_diagrams` ジョブで自動実行されるが、**ローカルでの即時確認が必要な場合**は以下を実行すること：

```bash
bash scripts/generate-mermaid-diagrams.sh
```

- 生成されたPNGは `docs/diagrams/` に格納される（PDF納品物への埋め込み用）

## セキュリティ要件（受託開発）

- 設計書内にパスワード・APIキー・トークンなどの実値を記載しない
- テストデータの例示には実在する個人情報を使用しない
- セキュリティに関わる設計判断は `docs/guides/security.md` に明記すること

## テンプレートの使用（必須）

仕様書を新規作成する場合、必ず `docs/templates/` 配下の該当テンプレートを参照し、そのフォーマット・セクション構造に従うこと。

| ドキュメント種別 | テンプレート                              |
| ---------------- | ----------------------------------------- |
| API仕様書        | `docs/templates/spec-api-template.md`     |
| DB設計書         | `docs/templates/spec-db-template.md`      |
| 要件定義書       | `docs/templates/requirements-template.md` |

## フォーマットルール（表・図・コードの統一）

- エンドポイント一覧・テーブル一覧は必ず**表形式（Markdownテーブル）**で記述
- パラメータ定義は必ず **型・必須/任意・説明** を列に持つ表で記述
- カラム定義は必ず **型・制約・説明** を列に持つ表で記述
- JSON例は必ず ` ```json ` コードブロックで記述
- ER図やフロー図は ` ```mermaid ` コードブロックで記述
- 各設計書の冒頭にYAMLフロントマター（title, project, version, date, author）を含めること

## MarkdownLint準拠と自動フォーマット（CI対策・必須）

- ドキュメントを新規作成・更新した後は、**必ず** `npx markdownlint-cli "docs/**/*.md" --fix` を実行すること。
- **AIが頻発させるLintエラーと対策:**
  - `MD040`: バッククォート3つのコードブロックを書く際は、**必ず**言語名（`text`, `json`, `css`, `bash` 等）を指定すること。
  - `MD036`: 見出し（`#` や `##`）の代わりに、**太字**（`**太字**`）だけの行を作らないこと。

## PDF変換時のメタデータ

顧客提出用PDFに変換する前提で、以下を各設計書の冒頭に記述すること：

```yaml
---
title: "ドキュメントタイトル"
project: "GROWTH MILE"
version: "1.0"
date: "YYYY-MM-DD"
author: "作成者名"
---
```
