# バックエンド堅牢性

> [INFO] 汎用コアルール。API・データ層を持つすべてのプロジェクトに適用する。

## 入力バリデーション（信頼境界）

- 外部から入る全データ（HTTP リクエスト・環境変数・外部 API レスポンス・キュー・ファイル）は、信頼境界で実行時スキーマバリデーションを行う
- 型システムはコンパイル時のみ有効。実行時検証を省略しない
- バリデーション失敗は 4xx で即時返却し、部分的に処理を進めない

## エラーハンドリング

- カスタムエラークラスを使用する。生の `Error` を投げない
- 空 catch ブロック・例外の握りつぶしは禁止。捕捉したら「回復する / 変換して再送出する / ログして通知する」のいずれかを必ず行う
- API は適切な HTTP ステータスコードと構造化 JSON エラーを返す。内部実装の詳細・スタックトレースをクライアントへ漏らさない
- 予期されるエラー（バリデーション・404 等）と予期しないエラー（バグ・障害）を区別して扱う

## トランザクション・整合性

- 複数の書き込みが「すべて成功 or すべて失敗」であるべき場合は、必ず 1 トランザクションにまとめる
- トランザクション内で外部 API 呼び出し・長時間処理を行わない
- 更新系 API は冪等に設計する（リトライ・二重送信で壊れない）
- 同時更新の競合が起きうる箇所は、楽観ロック（バージョン列）等の方針を設計時に決めて DB 設計書に記録する

## 外部通信の堅牢性

- 外部 API 呼び出しには必ずタイムアウトを設定する
- リトライは冪等な操作のみ、指数バックオフ + 上限回数付きで行う
- 失敗が連鎖する箇所はフォールバック・サーキットブレーカーの要否を検討する

## データアクセス

- N+1 クエリを作らない（一覧 + 関連取得は JOIN / include / バッチ取得で解決）
- 一覧 API は必ずページネーションを実装する（無制限の全件取得禁止）
- インデックスは検索・結合条件に合わせて設計し、DB 設計書に記録する

## ログ・可観測性

- ログには「いつ・どこで・何が・どのリクエストで」を含める（リクエスト ID 等の相関情報）
- 個人情報・秘密情報（メール・パスワード・トークン）をログに出さない
- エラーログには根本原因の調査に足る情報（入力の概要・処理状態）を含める
