Node.js のインストール

GROWTH MILEは Node.js 20.x LTS 系を使用します。バージョン違いに備えて nvm 経由のインストールを推奨します。

Windows (nvm-windows)
# https://github.com/coreybutler/nvm-windows/releases # nvm-setup.exe をダウンロードして実行 nvm install 20 nvm use 20 node -v # v20.x.x npm -v
macOS (nvm)
# nvm のインストール curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.40.1/install.sh | bash # ターミナルを開き直してから nvm install 20 nvm use 20 node -v
注意すでに別バージョンのNode.jsが入っている場合は、いったんアンインストールしてからnvm経由で入れ直すと事故が減ります。

SECTION 1 — 6/9

SSH キーを作って GitHub に登録

毎回パスワードを入れずにGitHubと通信するため、SSH鍵を作成して登録します（Windows/Mac共通）。

ターミナル / PowerShell
# 1. 鍵を生成（メールはGitHubアカウントのもの） ssh-keygen -t ed25519 -C "you@example.com" # 保存場所はEnterでデフォルトでOK、パスフレーズも任意 # 2. 公開鍵を表示してコピー # macOS / Linux: cat ~/.ssh/id_ed25519.pub # Windows (PowerShell): type $env:USERPROFILE\.ssh\id_ed25519.pub # 3. GitHub の Settings → SSH and GPG keys → New SSH key に貼り付け # 4. 接続テスト ssh -T git@github.com # "Hi xxx! You've successfully authenticated..." と出れば成功
絶対に公開しないこと登録するのは id_ed25519.pub（公開鍵）です。id_ed25519（秘密鍵、拡張子なし）は誰にも見せず、SlackやGitHubに貼らないこと。

SECTION 3 — 2/4

コミット規約 (Conventional Commits)

コミットメッセージは <type>: <summary> の形式で書きます。プロジェクトでは commitlint がチェックします。

type 意味 例
feat 新機能追加 feat: add user profile page
fix バグ修正 fix: handle null session in middleware
docs ドキュメント docs: update README setup section
style 整形のみ（動作変更なし） style: format with prettier
refactor リファクタ refactor: extract auth helper
test テスト追加・修正 test: add login flow e2e test
chore その他雑務 chore: bump next to 16.2.2
commitlintで弾かれますcommitフック (husky) によりフォーマットを満たさないと git commit が失敗します。慣れるまでは少しめんどうですが、確実にチームの履歴がきれいになります。

練習: 規約に沿っているのはどれ？

次のコミットメッセージで、規約OKなものを選んでください

A.

Updated login form

答え

NG — typeが無く動詞の過去形。feat: add login form validation のように書く。

B.

feat: add password reset email flow

答え

OK — typeが正しく、変更内容が一目で分かる短い要約になっている。

C.

fix:バグ修正

答え

NG — typeとsummaryの間に半角スペースが必要。要約が抽象すぎる。fix: handle 401 on session refresh など具体に。

D.

chore: bump prisma to 7.7.0

答え

OK — 依存関係更新は chore でOK。バージョンが具体的なのも良い。

SECTION 4 — 1/4

コーディング規約のキホン

命名規則
  • 関数・変数は camelCase（例: getUserById）
  • コンポーネント・クラスは PascalCase（例: UserProfileCard）
  • 定数は UPPER_SNAKE_CASE（例: MAX_RETRY）
  • ファイル名はkebab-caseまたはPascalCaseでコンポーネントと合わせる
型・モジュール規約
  • any 型は禁止。どうしても必要なら unknown + 型ガード
  • "use client" は必要最小限。Server Componentがデフォルト
  • Zodで入力スキーマを定義し、型を z.infer で導出
  • UI状態は zustand を使う（グローバルにReact Contextは使わない）
規約の正本このスライドは概要です。詳細は AGENTS.md と .claude/rules/ 配下のドキュメントを必ず確認してください。

どこに何を置く？ — ディレクトリ構造

プロジェクトルート
GrowthMile/ ├── src/ │ ├── app/ # App Router: pages, layouts, API routes │ ├── components/ # 共有Reactコンポーネント │ ├── features/ # 機能別モジュール (UI + ロジック) │ └── lib/ # Prisma, NextAuth, 共通ユーティリティ ├── prisma/ # schema.prisma, マイグレーション ├── public/ # 静的ファイル ├── e2e/ # Playwright (workspace外) ├── docs/ # 要件定義・設計書 ├── scripts/ # ユーティリティスクリプト └── .github/ # Actions, PRテンプレ, CODEOWNERS
迷ったら features/ か components/ 単一機能でしか使わない部品 → features/<feature>/。複数機能で再利用する部品 → components/。境界に迷ったらレビューで相談。
SECTION 4 — 3/4

Lint / Format / Test の回し方

PRを出す前に、ローカルでひととおり通すことが最低ライン。CIで弾かれるとレビュアの時間を奪います。

PR前のセルフチェック（コピペ用）
# 1. フォーマット npm run format # 2. Lint npm run lint # 3. ユニットテスト npm run test # 4. ビルドが通るか npm run build # 5. E2E（変更が大きい時） cd e2e && npx playwright test
husky + lint-stagedコミット時に変更ファイルだけPrettier/ESLintが自動実行されます。コミットが失敗した場合は、メッセージを読んで修正してから再コミットしてください。
テストの passWithNoTests新規機能を追加した際は、ユニットテスト or E2Eを最低1本書く方針。「テストゼロでも通る」設定はあくまで暫定です。
SECTION 4 — 4/4

トラブルシューティング FAQ

npm install が失敗する

Node.jsのバージョンを確認 (node -v)。20.x でなければ nvm use 20。それでもダメなら rm -rf node_modules package-lock.json && npm install。

docker compose up に失敗する

Docker Desktopが起動しているか確認。ポート5432が他のPostgreSQLで使われていないか lsof -i :5432（Mac）/ netstat -ano | findstr 5432（Win）で確認。

prisma migrate でエラー

.env の DATABASE_URL が正しいか、DBコンテナが healthy か確認。スキーマ衝突の場合は npx prisma migrate reset（ローカル限定）。

git commit が husky で弾かれる

エラーログをよく読む。Conventional Commits 形式を満たしていない、もしくはESLint/Prettierがエラーを出している。npm run format && npm run lint で自動修正してから再コミット。

ssh -T git@github.com で permission denied

公開鍵がGitHubに登録されていない可能性大。GitHub Settings → SSH keys を確認。鍵のファイル名がデフォルト (id_ed25519) でない場合は ~/.ssh/config での設定が必要。

解決しないとき

ためらわず Slack の #dev-help 等で質問を。「何をしたら」「何が起きたか」「どう試したか」をセットで書くと解決が早いです。
